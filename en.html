ADR-01 · THN (The Hotels Network) In production
Event Sourcing with snapshots to guarantee traceability per bounded context
Context

Multiple bounded contexts needed a reliable history of business changes and the ability to reconstruct state at any point in time, without penalizing read performance as event volume grew.

Decision

Event store with periodic snapshots + CQRS, splitting command, query and event buses with Symfony Messenger. Cross-context streaming via Kafka; AWS SQS with dead-letter queues for resilience in async consumption.

Outcome

Auditable state reconstruction for any entity, real decoupling between contexts, and fewer "why don't the numbers match" incidents — the system explains for itself how a given state was reached.

ADR-02 · THN (The Hotels Network) In production
Claude Code agents for convention auditing and code review
Context

Keeping team conventions consistent (PHPStan level 9, PSR-12, DDD/hexagonal patterns) was becoming increasingly costly to review manually as the codebase and team grew.

Decision

Designed specialized skills and agents with Claude Code, with MCP integrations, to automate code review, convention auditing and data diagnostics before reaching human review.

Outcome

Less team time spent on repetitive style/convention reviews; human reviews focused on design and business logic rather than style.

ADR-03 · Promofarma Completed
CQRS + Event Sourcing in a high-volume system
Context

Complex business domains with high transaction volume needed to clearly separate reads and writes to maintain consistency without query bottlenecks.

Decision

Domain modeling with Domain-Driven Design and CQRS implementation, with asynchronous service-to-service communication via Kafka within a microservices-oriented architecture.

Outcome

Guaranteed data traceability, with reads and writes scaled independently.

ADR-04 · Atrápalo / Drinks and Co Completed
Progressive monolith refactor towards Domain-Driven Design
Context

PHP 5 codebases, with business logic tightly coupled to infrastructure and framework code, with no clear boundaries between domains, made the product increasingly difficult to evolve and maintain as it grew.

Decision

Progressive process carried out alongside the PHP version migration (5 → 7 → 8): extracting business logic from infrastructure, identifying the natural bounded contexts within the monolith, and applying best practices and clean code (strict typing, removing dead code, reducing coupling) in each migrated module.

Outcome

Monolith incrementally transformed into a more domain-oriented, modular architecture, on a modern and maintainable codebase, without service downtime or a full rewrite.
